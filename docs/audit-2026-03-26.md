## Problemas Críticos

### 🔴 C1 — Divergência de tipo `PrioridadeGargalo`: `'alta'|'media'|'baixa'` vs `'URGENTE'|'ALTA'|'NORMAL'|'MONITORAR'`
**Arquivo**: `src/types/gargalo.ts` + `src/utils/analisarGargalo.ts`
**Impacto**: `PrioridadeGargalo` está tipado como `'alta' | 'media' | 'baixa'`, mas o CLAUDE.md e o tipo `PrioridadeDiligencia` usam `'URGENTE' | 'ALTA' | 'NORMAL' | 'MONITORAR'`. O mapeamento em `criarDiligenciaDeGargalo.ts` compensa, mas o TypeScript não captura o desacordo real: `prioridadePorDias()` em `analisarGargalo.ts` retorna literais que não batem com nenhum tipo externo estável. Se `PrioridadeGargalo` for atualizado sem atualizar `analisarGargalo.ts`, o map silencia o erro.
**Fix**:
```ts
// src/types/gargalo.ts
export type PrioridadeGargalo = 'URGENTE' | 'ALTA' | 'NORMAL' | 'MONITORAR'

// src/utils/analisarGargalo.ts — atualizar prioridadePorDias
function prioridadePorDias(dias: number): PrioridadeGargalo {
  if (dias > 30) return 'URGENTE'
  if (dias > 15) return 'ALTA'
  return 'NORMAL'
}

// src/utils/criarDiligenciaDeGargalo.ts — PRIORIDADE_MAP vira identidade
const PRIORIDADE_MAP: Record<PrioridadeGargalo, PrioridadeDiligencia> = {
  URGENTE: 'URGENTE', ALTA: 'ALTA', NORMAL: 'NORMAL', MONITORAR: 'MONITORAR',
}
```

### 🔴 C2 — `diligencia.service.ts` ainda é localStorage-only (TODO não cumprido)
**Arquivo**: `src/services/diligencia.service.ts:1`
**Impacto**: O CLAUDE.md documenta "migração automática one-shot" e API REST em `/api/diligencias`, mas o arquivo real contém apenas funções síncronas de `localStorage`. `DashboardOperacional`, `FilaDiligencias`, `Navigation` e `ProcessDetail` chamam essas funções sem `await`. Se o backend já expõe os 5 endpoints de diligências, toda a camada de persistência está desconectada. Dados não persistem entre dispositivos, nenhum usuário diferente vê a fila, e a migração Supabase (migration 003) fica orphan.
**Fix**: Reescrever `diligencia.service.ts` para usar `apiClient` apontando para `/api/diligencias`, com fallback localStorage e migração one-shot conforme documentado no CLAUDE.md. Todos os callers precisarão se tornar `async`.

## Problemas Altos

### 🟠 A1 — `DashboardOperacional`: `useState` inicializado com chamada síncrona que mascara async futuro
**Arquivo**: `src/pages/DashboardOperacional.tsx:~30`
**Impacto**: `useState<DiligenciaOperacional[]>(() => listarDiligencias())` usa inicializador lazy síncrono. Quando C2 for corrigido e `listarDiligencias` virar async, este pattern silenciosamente ficará com a lista vazia no estado inicial (a função retornará uma Promise, não um array). Não há `useEffect` de carregamento nem tratamento de loading/erro.
**Fix**:
```tsx
const [lista, setLista] = useState<DiligenciaOperacional[]>([])
const [loading, setLoading] = useState(true)

useEffect(() => {
  listarDiligencias().then(setLista).finally(() => setLoading(false))
}, [])
```

### 🟠 A2 — `error: any` implícito em handlers de evento no backend (`callMCPTool` catch)
**Arquivo**: `backend-server.js` — bloco `catch (error)` em `callMCPTool` e na rota `/api/process/visao-geral`
**Impacto**: O código acessa `error.message`, `error.message?.includes(...)` sem verificação de tipo. Em Node.js qualquer valor pode ser thrown. Se um não-Error for lançado (ex: string, number), `error.message` é `undefined`, causando log incorreto e potencialmente mascarando o erro real. Já ocorreu workaround com `error.message?.includes(...)` mas não de forma sistemática.
**Fix**:
```js
const msg = error instanceof Error ? error.message : String(error)
```
Aplicar em todos os blocos `catch` que acessam `error.message` diretamente.

### 🟠 A3 — `mcp.service.ts`: `Record<string, any>` e `error: any` explícito
**Arquivo**: `src/services/mcp.service.ts:~15,~35`
**Impacto**: `callBackendAPI(endpoint: string, data: Record<string, any> = {})` usa `any` no parâmetro de dados, e o bloco catch usa `catch (error: any)`. Viola a regra "TypeScript strict — sem `any`" do CLAUDE.md.
**Fix**:
```ts
async function callBackendAPI(endpoint: string, data: Record<string, unknown> = {}) { ... }
// catch:
} catch (error) {
  const msg = error instanceof Error ? error.message : String(error)
```

### 🟠 A4 — `Navigation`: polling não está protegido por autenticação; `listarAlertas` chama API sem verificar `isAuthenticated`
**Arquivo**: `src/components/layout/Navigation.tsx:~15`
**Impacto**: O `useEffect` que chama `listarAlertas()` e `listarDiligencias()` a cada 60s não verifica `isAuthenticated` (o hook `useAuth` está importado mas não é claro se é usado para este guard). Se o usuário não estiver autenticado e o backend retornar 401, o `console.warn` suprime o erro, mas o polling continua fazendo 2 requests/minuto desnecessários.
**Fix**:
```tsx
const { isAuthenticated } = useAuth()
useEffect(() => {
  if (!isAuthenticated) return
  // ... setup interval
}, [isAuthenticated])
```

## Problemas Médios

### 🟡 M1 — `CacheTimestamp`: `useEffect` com `setInterval` sem verificação se o intervalo já foi limpo antes de criar novo
**Arquivo**: `src/components/common/CacheTimestamp.tsx`
**Impacto**: O componente cria um `setInterval` dentro do `useEffect` de atualização de `minutesAgo`. O cleanup (`clearInterval`) só ocorre no retorno do `useEffect`. Se `timestamp` muda frequentemente, há janela para múltiplos intervals simultâneos até o cleanup ser chamado.
**Fix**: Armazenar o ID em `useRef` e limpar antes de criar:
```ts
const intervalRef = useRef<ReturnType<typeof setInterval> | null>(null)
// dentro do useEffect:
if (intervalRef.current) clearInterval(intervalRef.current)
intervalRef.current = setInterval(updateTime, 60_000)
return () => { if (intervalRef.current) clearInterval(intervalRef.current) }
```

### 🟡 M2 — `ProcessDetail`: `listarDiligenciasPorCNJ` chamada síncronamente em `useState` — mesma vulnerabilidade de C2/A1
**Arquivo**: `src/pages/ProcessDetail.tsx` — inicialização do estado `diligencias`
**Impacto**: Idêntico ao A1. Quando `diligencia.service.ts` for corrigido (C2), este estado quebrará silenciosamente. A aba de Diligências mostrará badge 0 permanentemente.
**Fix**: Mover para `useEffect` com `await` assim como `verificarCadastro`.

### 🟡 M3 — `FilaDiligencias`: `listarDiligencias` chamada em `useEffect` sem `await` (função síncrona agora, assíncrona depois de C2)
**Arquivo**: `src/pages/FilaDiligencias.tsx` — `useEffect` de carregamento inicial
**Impacto**: O `useEffect` provavelmente chama `setLista(listarDiligencias())` diretamente. Após C2, isso quebrará da mesma forma que A1/M2. Deve ser estruturado como async desde já para evitar retrabalho.
**Fix**: Usar pattern `useEffect(() => { let cancelled = false; listarDiligencias().then(d => { if (!cancelled) setLista(d) }); return () => { cancelled = true } }, [])`.

---

### 🟡 M4 — `process.ts`: `jsonResumo?: Record<string, any>` e `Process.id?: string` opcionais sem discriminante
**Arquivo**: `src/types/process.ts`
**Impacto**: `jsonResumo` usa `any` no tipo genérico. Além disso, `Process.id` é opcional enquanto o objeto é usado em múltiplos lugares com acesso direto a `id` sem verificação de nulidade. Preferível tipagem mais estrita.
**Fix**: `jsonResumo?: Record<string, unknown>` e documentar que `id` só é `undefined` antes de persistência.

---

### 🟡 M5 — `MeusProcessos`: `handleRemover` e outros handlers de evento assíncronos não são `useCallback`
**Arquivo**: `src/pages/MeusProcessos.tsx:~70`
**Impacto**: `handleRemover`, `handleMonitorar`, `handleMonitorarTodos` são funções async definidas inline na função do componente sem `useCallback`. São passadas implicitamente via closures para handlers de botão. Cada render recria essas funções, causando re-renders desnecessários se passadas como props.
**Fix**: Envolver em `useCallback` com as dependências corretas (`carregar`, `showToast`, `setMonitorando`).

## Problemas Baixos

### 🔵 B1 — `backend-server.js`: `audit()` fire-and-forget sem `.catch()`
**Arquivo**: `backend-server.js` — função `audit()`
**Impacto**: O `.then(({ error }) => ...)` trata erros Supabase corretamente, mas se a própria Promise da chamada `.insert()` rejeitar (ex: network error), há um unhandled rejection silenciosa.
**Fix**: Adicionar `.catch(err => console.warn('[audit] Rejeição inesperada:', err.message ?? err))` após o `.then`.

**Arquivo**: `src/services/escritorio.service.ts` — blocos `catch`
**Impacto**: Padrão similar ao A2/A3. Os blocos catch acessam propriedades de `error` sem verificação de tipo.
**Fix**: Aplicar `error instanceof Error ? error.message : String(error)` consistentemente.

### 🔵 B3 — `src/utils/analisarGargalo.ts`: regra `PARADO_GENERICO` usa literal `'media'` hardcoded (não vai capturar erros de tipo após C1)
**Arquivo**: `src/utils/analisarGargalo.ts:~35`
**Impacto**: A linha `prioridade: 'media'` no bloco `PARADO_GENERICO` é um literal de string que não passa pelo `prioridadePorDias()`. Após corrigir C1, o TypeScript vai reclamar desse literal isolado.

### 🔵 B4 — `Navigation.tsx`: `navLink` criada como função inline no corpo do componente sem `useCallback`
**Arquivo**: `src/components/layout/Navigation.tsx:~22`
**Impacto**: `navLink` é recriada a cada render. Como é uma função interna que retorna JSX, não é passada como prop para filho, portanto o impacto real é mínimo. Mas é inconsistente com o padrão adotado no restante do projeto.
**Fix**: Extrair para fora do componente como função pura (não precisa de estado) ou envolver com `useCallback`.

## Observações Gerais

1. **Padrão de error handling no backend** está bem implementado na maioria das rotas Supabase (não expõe `error.message` ao cliente). A violação de A2 é localizada no bloco MCP e em acessos a `error.message` em `catch` genéricos.
2. **CORS** está correto: `['GET', 'POST', 'PUT', 'DELETE']` cobre todos os métodos usados, incluindo os novos endpoints de diligências.
3. **Rate limiting** em 3 camadas está corretamente aplicado.
4. **Imports via `@/`** são usados consistentemente em todo `src/`.
5. **React Query hooks** têm `staleTime` e `enabled` corretos conforme CLAUDE.md.
6. **`useMemo`/`useCallback`** estão bem aplicados nos componentes principais (`ProcessDetail`, `MeusProcessos`). As ausências reportadas (M5, B4) são casos secundários.
